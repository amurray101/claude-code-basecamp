Session Management Cheatsheet

When to reach for each command — not what they are, but when

/compact

When: Claude starts repeating itself, forgetting earlier context, or the session has been running 20+ minutes on a complex task.

/clear

When: You've hit a dead end, the conversation is tangled, or you're switching to a completely different task.

/cost

When: A customer asks "how much does this cost?" Run it live. Showing real spend builds trust.

Plan Mode

When: A skeptical audience wants to see the reasoning without changing any files. Safe, impressive, trust-building.

[TRANSITION] "You know what these commands do from the foundations. This is about when to reach for each one."

[DEMO TIP] Demo /cost live during a session. Customers love seeing the real-time spend tracker. It immediately defuses the "costs will spiral" objection.

Plan Mode is especially powerful for first demos with security-conscious teams. "Let me show you how Claude would approach this" — without actually changing any files.

When Things Go Wrong

You will see these in the lab. Here's how to recover.

Kitchen Sink

Pattern: Too many unrelated tasks in one session. Context fills, quality drops.

Recovery: /clear and start fresh. One task, one session.

Correction Spiral

Pattern: Correcting Claude 5+ times. The conversation gets tangled. Each correction makes it worse.

Recovery: Stop correcting. Start a new session with a clearer prompt.

Over-Specified CLAUDE.md

Pattern: 200+ lines of rules. Claude follows bad rules rigidly instead of using judgment.

Recovery: Cut to 30–80 lines. Conventions, not micromanagement.

Context Exhaustion

Pattern: Vague prompts like "improve this codebase." Claude reads everything, accomplishes nothing.

Recovery: Be specific. "Refactor the auth module" beats "improve the code."

[TRANSITION] "In customer engagements, you will see these patterns. Knowing how to diagnose and recover from them is what makes you credible."

[KEY MOMENT] These are not theoretical. Students will hit at least one of these during the lab. When they do, point them back to this slide. The recovery patterns should become muscle memory.

Hooks: The Compliance Demo

"When a CISO asks how you control Claude, show them this."

.claude/hooks.json
{
"pre-commit": [
"npm run lint",
"npm run type-check",
"npm test -- --bail"
],
"post-edit": [
"npm test -- --related --bail"
]
}

The punchline

If the hook fails, the action is blocked. Period. Claude cannot bypass these gates. They are enforced at the system level, not the prompt level.

The demo moment

Introduce a deliberate bug. Try to commit. Watch the hook catch it. "These are enforced gates, not suggestions."

[DEMO TIP] The word "enforced" matters. Say it: "These are enforced gates." CISOs hear "enforced" and relax. They hear "suggested" and tense up.

Security: Defense-in-Depth

The layered answer to "How do I know Claude won't..."

1 Sandbox OS-level filesystem isolation. Claude cannot access files outside the project directory.
2 Permissions Tool-level controls. Approve/deny file writes, bash commands, network access per action.
3 Hooks Team-level guardrails. Pre-commit checks that Claude cannot bypass.
4 Managed Settings Admin-level policies. Org-wide restrictions on model access, data handling, and tool usage.

Layer by layer builds confidence. Start with sandboxing, end with managed settings.

[TRANSITION] "When Nova's CISO asks 'how do I know Claude won't exfiltrate patient data?' — this is your answer."

Walk through layer by layer: (1) Sandbox — OS-level isolation, Claude can't see files outside the project. (2) Permissions — developers approve or deny each action. (3) Hooks — team-enforced quality gates. (4) Managed settings — admin controls for the whole org.

The layered narrative builds confidence. Never just say "it's secure." Show the layers.

Objection Handling Frameworks

"It's not secure enough"

Walk the defense-in-depth stack: sandbox → permissions → hooks → managed settings. Then show the Trust Center. Layer by layer builds confidence.

"Costs will spiral"

Model routing: Haiku for automation, Sonnet for daily, Opus for hard problems. Usage dashboards and budget caps. Compare to the cost of an additional hire, not the cost of a Copilot seat.

"We already use Copilot"

They serve different needs. Copilot: line-level autocomplete. Claude Code: project-level agentic tasks. Teams use both. Ask: "What's the biggest refactor you've been putting off?"

"Will it replace our developers?"

Claude Code removes the parts of the job nobody likes: boilerplate, migration grunt work, chasing test failures across 50 files. What's left is the creative, high-judgment work. Teams report feeling more agency, not less.

[TRANSITION] "These are the four objections you will hear most. Practice the response until it's natural."

The security objection: layer by layer, not "it's secure." The cost objection: compare to a hire, not a seat. The Copilot objection: coexistence, not replacement. The job threat objection: reframe to what gets automated (boring stuff) vs. what stays (interesting work).
