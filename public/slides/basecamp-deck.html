Demo Narration Guide

Read

"Watch — it's scanning the existing routes, reading the test patterns, checking the package.json..."

Plan

"It's breaking this into steps — new route file, validation, tests, update the index..."

Act

"Now it's writing across multiple files — matching the conventions it found..."

Verify

"It's running the tests to check its own work. If something fails, it reads the error and fixes it."

[DEMO TIP] Practice saying these lines while the facilitator demos. Mouth the words. Get comfortable with the cadence. The narration should feel natural, not scripted. By the time you leave today, this should be second nature.

Review: Tools & Session Management

Plan Mode

Think before acting. Claude analyzes and proposes a plan — without changing files. Review, give feedback, then green-light execution.

Keyboard Shortcuts

One job, one session. Use /compact every 15–20 min on large codebases. /clear between unrelated tasks. /cost after a demo task to show ROI.

Two Traps You'll Hit in the Lab

The two most common mistakes when writing a CLAUDE.md for the first time

Over-Specified CLAUDE.md

The trap: You write 200+ lines of rigid rules. Claude follows them all — including the bad ones — instead of using judgment. More rules does not mean better output.

Recovery: Cut to 30–80 lines. Conventions and patterns, not micromanagement.

200+ lines → 30–80

Context Exhaustion

The trap: You give a vague prompt like “clean up this codebase.” Claude reads everything, accomplishes nothing, and burns through your context window.

Recovery: Be specific. “Refactor the user module to use async/await” beats “improve the code.”

vague → specific
