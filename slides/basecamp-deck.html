What Is CLAUDE.md?

A markdown file that tells Claude how your team works

CLAUDE.md
# Lumen Logistics API

## Architecture
Node.js monorepo. Services in /services,
shared libs in /packages. Each service has
its own package.json.

## Conventions
- Use async/await, never callbacks
- Imports: node builtins, then external,
  then internal (alphabetical within each)
- Error handling: custom AppError class

## Testing
Run tests: npm test -- --bail
Framework: Jest. One test file per module.
Naming: *.test.ts next to source file.

Plain markdown

No special syntax. Just headings, bullets, and prose. Anyone on the team can read and edit it.

Human-authored

You write it, not Claude. It captures what your senior engineers know but haven't documented. Claude reads it at session start.

Checked into git

Lives in the repo root. Versioned, reviewed in PRs, shared by every developer using Claude Code on this project.

[TRANSITION] "This is what a real CLAUDE.md looks like. Plain markdown — no special syntax, no config format. You write it by hand, capturing the conventions your team already follows."

[KEY POINTS] Three things to emphasize: (1) It's plain markdown, not YAML or JSON — anyone can edit it. (2) Humans write it, Claude reads it — it's your instructions to Claude, not the other way around. (3) It lives in the repo and goes through code review like any other file.

[COMMON QUESTION] "Can Claude write its own CLAUDE.md?" Yes, you can ask Claude to draft one based on reading the codebase — but a human should review and edit it. The value is encoding human judgment, not automated guesses.

Prompt Patterns That Work

The difference between a good prompt and a great one: specificity and structure

Vague — Claude guesses

“Add a shipments endpoint.”

Structured — Claude follows

“Add a new POST /api/shipments endpoint that creates a shipment record. Follow the same patterns as the existing routes in our CLAUDE.md.”

The two-part pattern

WHAT — the task you want done
+
HOW — the conventions and patterns to follow

[TRANSITION] "CLAUDE.md gives Claude the conventions. But your prompt is what activates them. A vague prompt forces Claude to guess even with a perfect CLAUDE.md. A structured prompt gets consistent output on the first try."

[KEY POINT] The WHAT + HOW pattern is the single most useful prompt structure for agentic coding. WHAT is the task. HOW is which conventions to follow, which patterns to match, which quality bar to hit. Vague prompts force Claude to guess. Specific prompts get convention-matching output on the first try.

[EXERCISE CUE] Ask the room: "Take the vague prompt on screen. How would you make it specific?" Get 2-3 answers. Compare them to the structured version. The goal is for trainees to internalize this pattern before the lab.

Session Hygiene

Three commands and one rule that keep your sessions productive

/compact

Summarizes the conversation and frees context space. Use proactively every 15–20 min — don't wait until Claude starts forgetting.

/clear

Resets the entire context. Use when switching tasks — different tasks need fresh context.

/cost

Shows tokens consumed and total spend. Run after a real task in customer demos: “That refactor cost $0.08. Your developer would have spent 45 minutes.”

The one-job-one-session rule

Don't cram unrelated tasks into one session. Refactor a module, then add a feature, then fix a bug in a different file — quality drops after the third task. The context fills with unrelated code. For complex work: Plan Mode to scope it, then a fresh session to execute each piece.

[TRANSITION] "Three commands and one habit. /compact is your pressure release valve. /clear is your fresh start. /cost is your sales weapon. And the one-job-one-session rule is what separates power users from frustrated users."

[KEY POINT] Teach customers to use /compact proactively, not reactively. If you wait until Claude starts forgetting, you've already lost context. Compact every 15-20 minutes in long sessions.

[SALES TIP] /cost is your friend in customer conversations about pricing. Run it after a real task during a demo: "That refactoring task — reading the codebase, planning changes, writing code, running tests — cost $0.08. Your developer would have spent 45 minutes." Let the math speak for itself.

[KITCHEN SINK PATTERN] The one-job-one-session rule addresses the "kitchen sink" anti-pattern. If trainees ask about it, explain: multiple unrelated tasks in one session means Claude's context fills with code from task 1 that's irrelevant to task 3. Quality degrades. The fix is simple: /clear between tasks, or start a new session.

Two Traps You'll Hit in the Lab

The two most common mistakes when writing a CLAUDE.md for the first time

Over-Specified CLAUDE.md

The trap: You write 200+ lines of rigid rules. Claude follows them all — including the bad ones — instead of using judgment. More rules does not mean better output.

Recovery: Cut to 30–80 lines. Conventions and patterns, not micromanagement.

200+ lines → 30–80

Context Exhaustion

The trap: You give a vague prompt like “clean up this codebase.” Claude reads everything, accomplishes nothing, and burns through your context window.

Recovery: Be specific. “Refactor the user module to use async/await” beats “improve the code.”

vague → specific

[TRANSITION] "Two traps. You will hit at least one. Over-specifying your CLAUDE.md is #1 — you'll want to add everything you find. Resist. Context exhaustion is #2 — scope your prompts tightly."

[NOTE] There are other failure patterns (Kitchen Sink, Correction Spiral) covered in the reference materials. These two are the ones that matter for today's lab.

Beyond Lumen: Where Else CLAUDE.md Can Help

The same CLAUDE.md playbook applies whenever a team has unwritten rules

Legacy migration

A team rewriting Java to Kotlin, or Rails to Node. The old patterns are everywhere but the new conventions aren't documented yet. CLAUDE.md encodes the target state so Claude writes new code, not more legacy.

Regulated industries

Healthcare, finance, government. Strict coding standards for audit trails, data handling, and naming. CLAUDE.md becomes an enforceable spec — not just a suggestion, but the input to hooks that block non-compliant code.

Acquisitions & merges

Two codebases, two sets of conventions, one repo. The acquiring team writes a CLAUDE.md for the merged codebase — instant consistency without rewriting everything by hand.

High developer turnover

Consultancies, agencies, teams with frequent rotation. Tribal knowledge walks out the door every quarter. CLAUDE.md captures it permanently and makes every new developer productive from day one.

Multi-team platform

A shared platform with 5+ teams contributing. Each team has subproject CLAUDE.md files. The platform team maintains the root file. Convention enforcement without micromanagement.

Open source maintainers

External contributors who don't know the project's style. A CLAUDE.md in the repo means every contributor using Claude Code automatically follows project conventions.

Security: Defense-in-Depth

The layered answer to "How do I know Claude won't..."

1 Sandbox OS-level filesystem isolation. Claude cannot access files outside the project directory.
2 Permissions Tool-level controls. Approve/deny file writes, bash commands, network access per action.
3 Hooks Team-level guardrails. Pre-commit checks that Claude cannot bypass.
4 Managed Settings Admin-level policies. Org-wide restrictions on model access, data handling, and tool usage.

Layer by layer builds confidence. Start with sandboxing, end with managed settings.

[TRANSITION] "When Nova's CISO asks 'how do I know Claude won't exfiltrate patient data?' — this is your answer."

Walk through layer by layer: (1) Sandbox — OS-level isolation, Claude can't see files outside the project. (2) Permissions — developers approve or deny each action. (3) Hooks — team-enforced quality gates. (4) Managed settings — admin controls for the whole org.

The layered narrative builds confidence. Never just say "it's secure." Show the layers.
