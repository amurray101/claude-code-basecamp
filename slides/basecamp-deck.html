The loop repeats. If the tests fail, Claude reads the error, plans a fix, acts on it, and verifies again. This self-correction is a powerful demo moment.

CLAUDE.md

The system prompt for your codebase

What it is

A markdown file at your project root that gives Claude Code persistent instructions: coding standards, architecture decisions, testing conventions, anything you would tell a new teammate on day one.

Why it matters

Without it, Claude reads more files, guesses at conventions, and often gets them wrong. With it, Claude starts faster, targets the right files, and matches your team's patterns.

The GTM angle

Writing the first CLAUDE.md together is a powerful onboarding moment with customers. It is the single most impactful configuration surface — every deployment should start here.

[TRANSITION] "CLAUDE.md is the most important file you will create in any customer engagement."

Think of it as a system prompt, but for an entire codebase. Every time Claude Code starts a session, it reads this file first. It shapes every decision Claude makes — what patterns to follow, what tools to use, what to check before committing.

CLAUDE.md in Practice

CLAUDE.md
# Project: Lumen Logistics API

## Architecture
- Express.js monorepo with route handlers in /src/routes/
- Shared utilities in /src/utils/ — never duplicate
- Tests live next to source: foo.ts → foo.test.ts

## Conventions
- TypeScript strict mode, async/await only
- All functions need JSDoc comments
- Imports: group by external, internal, types

## Testing
- Jest + supertest for API tests
- Every route: happy path + error case minimum
- Run `npm test` before committing

## Before committing
- Run `npm run lint && npm test`
- Never commit .env or node_modules

[DEMO] Walk through each section. "Architecture tells Claude where things live. Conventions tells Claude how to write. Testing tells Claude what to verify. Before committing tells Claude the quality gate."

Emphasize: this file should be 30-80 lines. NOT 200 lines. Over-specified CLAUDE.md files are a common anti-pattern — they constrain more than they help.

CLAUDE.md Hierarchy

Instructions cascade from general to specific

~/CLAUDE.md Personal defaults — applies to all your projects
/repo/CLAUDE.md Project root — coding standards, architecture, test patterns
/repo/src/CLAUDE.md Subdirectory — module-specific overrides
/repo/.claude/CLAUDE.md Team-specific — shared via git, enforced org-wide

More specific files override more general ones. Like CSS specificity for your codebase.

[TRANSITION] "CLAUDE.md isn't just one file. It's a hierarchy."

The key insight for enterprise customers: a root CLAUDE.md sets org-wide standards. Subdirectory CLAUDE.md files add module-specific rules. This maps to how real engineering orgs work — company-wide style guides with team-level exceptions.

For Solutions Architects: this hierarchy is how you design a CLAUDE.md strategy for a 200-person engineering org.

Hooks: The Compliance Answer

"How do I control what Claude does?" — every CISO, ever

{
"pre-commit": [
"npm run lint",
"npm run type-check",
"npm test -- --bail"
],
"post-edit": [
"npm test -- --related --bail"
]
}

Pre-commit

Enforced gates. Lint, type-check, and tests must pass before any commit. Not suggestions — requirements.

Post-edit

Runs tests on related files after Claude edits. Catches regressions in real time.

Security: Defense-in-Depth

The layered answer to "How do I know Claude won't..."

1 Sandbox OS-level filesystem isolation. Claude cannot access files outside the project directory.
2 Permissions Tool-level controls. Approve/deny file writes, bash commands, network access per action.
3 Hooks Team-level guardrails. Pre-commit checks that Claude cannot bypass.
4 Managed Settings Admin-level policies. Org-wide restrictions on model access, data handling, and tool usage.

Layer by layer builds confidence. Start with sandboxing, end with managed settings.

[TRANSITION] "When Nova's CISO asks 'how do I know Claude won't exfiltrate patient data?' — this is your answer."

Walk through layer by layer: (1) Sandbox — OS-level isolation, Claude can't see files outside the project. (2) Permissions — developers approve or deny each action. (3) Hooks — team-enforced quality gates. (4) Managed settings — admin controls for the whole org.

The layered narrative builds confidence. Never just say "it's secure." Show the layers.

"Will it replace our developers?"

Claude Code removes the parts of the job nobody likes: boilerplate, migration grunt work, chasing test failures across 50 files. What's left is the creative, high-judgment work. Teams report feeling more agency, not less.

[TRANSITION] "These are the four objections you will hear most. Practice the response until it's natural."

The security objection: layer by layer, not "it's secure." The cost objection: compare to a hire, not a seat. The Copilot objection: coexistence, not replacement. The job threat objection: reframe to what gets automated (boring stuff) vs. what stays (interesting work).
